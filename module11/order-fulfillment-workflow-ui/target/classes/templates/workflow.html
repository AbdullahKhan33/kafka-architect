UI-13: Order Fulfillment Workflow using Kafka Event Choreography

Kafka Event Choreography Flow

Each service listens to one topic and publishes the next event. This creates sequencing without one central workflow controller. 

ORDER_PLACED
    ↓
PaymentConsumer
    ↓
PAYMENT_RECEIVED
    ↓
PackingConsumer
    ↓
ORDER_PACKED
    ↓
ShippingConsumer
    ↓
ORDER_SHIPPED
    ↓
DeliveryConsumer
    ↓
DELIVERED

Important Clarification

Kafka transports and stores events. Kafka does not decide business rules like payment before packing.

The sequence is created by application design. PackingConsumer listens only to payment-events-topic. Therefore, packing starts only after a PAYMENT_RECEIVED event exists.

In this demo, orderId and correlationId are the same. In production, correlation ID may be a separate technical trace ID.

The trace dashboard is an application-level view. In production, this trace data should go to a database, Elasticsearch, OpenSearch, Azure AI Search, or an observability platform.
